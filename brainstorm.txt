Kintsugi Voice Biomarker Plugin — Objectives, Rationale, and System Design (v0.3, Real-Time, Privacy-First)

This system is designed as a plug-and-play integration layer that exposes Kintsugi Health’s open-source voice biomarker models (sourced via Hugging Face) to any voice-enabled platform while enforcing strict privacy constraints. The central requirement is that no sensitive user data—raw audio, derived features, or identifiable outputs—is ever stored at rest. All processing must occur in-memory, in real time, and results must be emitted immediately upon detection without persistence or replay capability. This reframes the system from a conventional ML inference service into a stateless, streaming signal processor with auditable guarantees around ephemerality and data minimization.

The primary objective is to enable real-time inference of mental health-related signals (e.g., depression likelihood, anxiety markers, stress indices) directly from live audio streams, while providing a developer-friendly interface that integrates seamlessly with existing voice infrastructures. Secondary objectives include supporting both hosted (SaaS) and self-hosted (Dockerized/on-prem) deployment models, abstracting model complexity behind stable APIs, and ensuring compatibility with widely used voice transport mechanisms such as WebRTC and Twilio Media Streams. The system prioritizes streaming-first processing, avoiding batch dependencies, and uses sliding window inference over short audio buffers (e.g., 5–10 seconds) that are continuously overwritten in memory.

Architecturally, the system consists of a voice input source (e.g., WebRTC browser stream, Twilio telephony stream, SIP, or mobile microphone input) feeding into an adapter layer responsible for normalizing diverse audio codecs into a standard format such as 16kHz mono PCM. This is followed by an ephemeral in-memory buffer that maintains a rolling window of audio data with no disk persistence. A lightweight preprocessing layer performs resampling, normalization, and any required feature extraction within strict latency constraints. The inference engine wraps Kintsugi’s Hugging Face models behind a stable interface, executing predictions on each audio window and producing structured outputs such as continuous scores and confidence values. These outputs are not stored; instead, they are immediately passed to an event emitter layer that streams results back to the client via WebSockets, gRPC, or similar real-time transport mechanisms. There is intentionally no API for historical retrieval, reinforcing the no-storage guarantee.

The developer-facing surface is provided through SDKs (initially TypeScript and Python) that abstract streaming, authentication, and event handling. A typical integration involves attaching the plugin client to a live audio stream and subscribing to emitted signal events in real time. The deployment model is hybrid: a hosted version enables rapid adoption with strict no-storage guarantees enforced at the service level, while a self-hosted version allows organizations (especially in healthcare contexts) to run the system داخل their own infrastructure for full data control and compliance alignment.

Privacy is treated as a first-class design constraint rather than an afterthought. Audio data is never written to disk, buffers are cleared immediately after inference cycles, and no training or model improvement is performed on user data. Logging is restricted to system-level metrics such as latency and error rates, explicitly excluding payloads, feature vectors, or user-linked outputs. For stronger guarantees, future iterations may enforce container-level restrictions (e.g., disabling filesystem writes entirely) to make persistence technically impossible rather than policy-based.

Integration priority is driven by industry usage patterns, with Twilio Media Streams and WebRTC-based browser applications targeted for the initial MVP, followed by SIP-based enterprise systems and eventually mobile SDKs. However, several technical risks remain. Hugging Face-hosted models may not be optimized for low-latency streaming inference and could require transformation (e.g., quantization or ONNX conversion). Latency budgets (<500ms end-to-end) may be challenged by preprocessing and model execution overhead. Audio variability across environments (noise, codecs, accents) may impact model reliability. Additionally, there is a non-trivial product risk around misinterpretation of outputs as clinical diagnoses, which necessitates careful API design and documentation.

Several critical questions must be resolved before proceeding further. Specifically, the exact input requirements of the Kintsugi models (minimum audio length, streaming compatibility, frame-level vs utterance-level inference) must be validated, as these directly determine feasibility of real-time processing. It must also be decided whether privacy guarantees should be enforced at a provable systems level (e.g., container sandboxing) versus best-effort engineering constraints. Another key decision is whether to prioritize edge/local inference to fully eliminate cloud exposure, or maintain a cloud-first approach with optional self-hosting. Finally, output semantics need to be defined: whether the system should emit raw continuous scores or only threshold-based alerts.

The immediate next step is to validate the Hugging Face model constraints and build a streaming-compatible inference wrapper. Without confirming that the underlying models can support low-latency, windowed inference, further system design risks being misaligned with actual capabilities.
